- 登录功能

- **AI智能瞭望**

因此，目前代码更像是一个“业务系统的基础骨架”，已经具备：

- 服务入口

## 3. 项目架构总览

项目采用典型的 **MVC 思路**，虽然没有使用传统大型框架的强约束结构，但职责分工已经很清晰。

### 3.1 MVC 对应关系

- **M（Model）模型层**
  - 负责数据访问、数据库操作、业务数据处理
  - 当前在 `app/models/` 下

- **V（View）视图层**
  - 负责页面模板渲染
  - 当前在 `app/templates/` 下

- **C（Controller）控制层**
  - 负责处理 HTTP 请求、参数校验、调用模型、返回页面或跳转
  - 当前在 `app/controllers/` 下

### 3.2 请求处理流程

当前系统的典型流程如下：

1. 用户从浏览器发起请求
2. Tornado 根据路由分发到对应 Handler
3. Handler 接收参数并进行校验
4. Handler 调用 Model 层进行数据库查询或写入
5. Handler 根据结果渲染模板或执行重定向
6. 浏览器接收 HTML 页面

## 4. 当前已实现功能

根据现有代码，项目已经完成以下功能。

### 4.1 登录功能

登录页由 `LoginHandler` 提供。

功能特点：

- 支持用户名和密码输入
- 空值校验
- 用户名密码错误提示
- 成功后写入 secure cookie
- 登录成功后跳转到首页

### 4.2 登录态验证

登录态由 `BaseHandler.get_current_user()` 统一处理。

特点：

- 从 secure cookie 中读取 `username`
- 若不存在则认为未登录
- 结合 Tornado 的 `@tornado.web.authenticated` 机制实现访问保护

### 4.3 首页访问控制

首页由 `IndexHandler` 提供，并加了认证装饰器：

- 未登录用户访问首页会被重定向到登录页
- 已登录用户可访问后台首页

### 4.4 退出登录

`LogoutHandler` 提供退出功能：

- 清除 cookie
- 跳转到登录页

### 4.5 用户存储与验证

用户数据存储在 SQLite 中，包含：

- 用户名
- 密码哈希
- 盐值
- 创建时间

密码不会明文保存，而是采用哈希 + 盐值方式。

## 5.3 `app/controllers/`

控制器层，负责接收请求、处理参数、调用模型、输出视图。

### `app/controllers/base.py`

作用：

- 定义基础控制器 `BaseHandler`
- 封装当前登录用户获取逻辑
- 为其他控制器提供统一基类

核心职责：

- 从 secure cookie 中读取用户名
- 返回当前登录用户或 `None`
- 支撑 `@tornado.web.authenticated`

### `app/controllers/auth.py`

作用：

- 认证相关控制器
- 处理登录、退出

主要包含：

- `LoginHandler`
- `LogoutHandler`

功能：

- GET：渲染登录页
- POST：校验用户名密码
- 登录成功：写入 secure cookie 并跳转首页
- 登录失败：返回错误信息
- 退出：清除登录状态

### `app/controllers/home.py`

作用：

- 主页/后台首页控制器

主要包含：

- `IndexHandler`

功能：

- 只有登录后才能访问
- 渲染后台首页模板
- 将当前用户名传递到模板中展示

### `app/controllers/__init__.py`

- 将 `controllers` 目录标记为 Python 包
- 文件中已写明控制器层职责约定

## 5.4 `app/models/`

模型层，负责与数据库交互和封装业务数据逻辑。

### `app/models/db.py`

作用：

- 提供数据库连接
- 定义数据库文件路径
- 初始化数据库表结构

核心内容：

- `_project_root()`：定位项目根目录
- `DB_PATH`：SQLite 数据库路径
- `get_connection()`：创建数据库连接，设置 `row_factory`
- `init_db()`：创建 `users` 表

### `app/models/user.py`

作用：

- 用户数据访问仓储层
- 封装用户创建和认证校验逻辑

核心能力：

- 创建用户时生成 salt
- 对密码做 PBKDF2 哈希
- 校验用户名密码是否匹配

注意：

- 当前代码体现了“密码安全存储”的意识
- 未来如果扩展用户体系，建议继续沿用“hash + salt”模式

### `app/models/__init__.py`

- 将 `models` 目录标记为 Python 包
- 说明中明确提到了 `db.py` 与 `user.py` 的职责

## 6. 数据库设计学习

当前数据库使用 SQLite，本地文件存储，适合轻量开发、调试和快速原型验证。

### 6.1 数据库位置

代码中数据库文件的最终路径为：

- `database/app.db`

### 6.2 当前数据表

#### `users`

字段：

- `id`：自增主键
- `username`：用户名，唯一，不为空
- `password_hash`：密码哈希，不保存明文密码
- `salt`：盐值
- `create_at`：创建时间，默认当前时间

### 6.3 数据库初始化流程

项目启动时会先执行：

- `init_db()`

作用：

- 自动创建 `users` 表
- 如果表已存在则忽略

这说明项目具有“开箱即用”的初始化特征。

## 7. 登录认证机制学习

这是当前项目最重要的已实现功能之一。

### 7.1 登录态保存

登录成功后，代码会：

- 使用 `set_secure_cookie("username", username)` 保存登录状态

说明：

- 不是普通 cookie
- 采用 Tornado 的 secure cookie 机制
- 更适合做认证态管理

### 7.2 当前用户获取

`BaseHandler.get_current_user()` 负责：

- 读取 secure cookie 中的 `username`
- 如果不存在返回 `None`
- 返回值供 `@tornado.web.authenticated` 使用

### 7.3 权限控制

首页通过 `@tornado.web.authenticated` 保护。

效果：

- 未登录无法访问首页
- 会自动跳转到 `login_url`

### 7.4 退出机制

退出时：

- `clear_cookie("username")`
- 回到登录页

## 9. 对后续“AI智能瞭望与智能问数系统”的开发建议

基于当前架构，后续开发建议遵循以下原则：

### 9.1 保持 Tornado MVC 思路

后续新功能尽量延续以下分层：

- `controllers/`：处理新请求
- `models/`：存储与业务逻辑封装
- `templates/`：页面展示
- `static/`：JS/CSS 资源

不要把所有逻辑都堆在 `app.py`。

### 9.2 新业务应按模块扩展

如果后续增加“智能瞭望”“智能问数”等功能，建议按功能划分控制器与模型，例如：

- `watch/` 相关控制器
- `query/` 相关控制器
- `dashboard/` 相关控制器
- `report/` 相关模型或服务逻辑

### 9.3 统一鉴权策略

未来所有核心业务页面建议都复用现有登录态机制，避免重复实现认证逻辑。

### 9.4 数据库设计要为扩展预留空间

如果后续要接入问数、日志、审计、智能分析记录、任务管理等功能，建议继续在 SQLite 结构上逐步扩展表设计，或者在合适时机迁移到更强的数据库。

### 9.5 前端页面建议分层清晰

随着系统扩大，建议区分：

- 基础布局模板
- 登录页面
- 仪表盘页面
- 列表页面
- 详情页面
- 图表页面
- 问答页面

这样可以减少模板重复。

## 11. 后续开发前建议重点确认的问题

在正式开始实现“AI智能瞭望与智能问数系统”之前，建议先明确以下内容：

### 11.1 功能范围

需要确认：

- 智能瞭望具体包含哪些页面和能力
- 智能问数的输入方式是什么
- 是否需要多轮问答
- 是否需要图表、报表、告警、订阅、历史记录

### 11.2 用户体系

需要确认：

- 是否只有单一管理员用户
- 是否需要角色权限
- 是否要支持多用户登录
- 是否要支持注册/找回密码

### 11.3 数据来源

需要确认：

- 问数的数据来源是什么
- 是否来自本地 SQLite
- 是否来自外部 API
- 是否来自上传文件
- 是否需要接入业务系统数据库

### 11.4 展现形式

需要确认：

- 纯页面展示
- 表格 + 图表
- 大屏可视化
- 交互式问答界面
- 实时刷新

### 11.5 AI 能力边界

需要确认：

- 是规则式问答还是大模型驱动
- 是否需要知识库
- 是否需要 RAG
- 是否需要对接第三方模型服务
